TypeParameterResolver

上文分析Reflector实现时，addGetMethod使用到了TypeParameterResolver.resolveReturnType(method, type)方法，我们来看下TypeParameterResolver是干什么的

重点关注上面框出来的public方法，这三个方法对外提供服务

  1. Type resolveFieldType(Field field, Type srcType)用于解析属性类型
  2. Type resolveReturnType(Method method, Type srcType)用于解析方法返回值类型
  3. Type[] resolveParamTypes(Method method, Type srcType)用于解析方法参数类型

三者实现方式类似，都使用了resolveType方法，我们以resolveFieldType为例

先看下getType() 和 getGenericType()的区别 ：
getType()： 获取属性声明时类型对象（返回class对象）
getGenericType() ： 返回属性声明的Type类型
1.首先是返回的类型不一样，一个是Class对象一个是Type接口。
2.如果属性是一个泛型，从getType（）只能得到这个属性的接口类型。但从getGenericType（）还能得到这个泛型的参数类型。
3.getGenericType（）如果当前属性有签名属性类型就返回，否则就返回 Field.getType()。

上面出现了三个类型判断，这三个类型又是什么呢？这三个类型均继承自Type接口

  • Class: 表示的是原始类型。Class类的对象表示JVM中的一个类或接口，每个Java类在JVM里都表现为一个Class对象。在代码中可以通过“类名.class”、“对象.getClass()”或“Class.forName(类名)”的方式获取。数组也会被映射成Class对象，所有类型相同且维数相同的数组都共享同一个Class对象。
  • ParameterizedType：表示的是参数化类型，例如List 、Map<Integer,String>这种带有泛型的类型。ParameterizedType接口中常用的方法有三个:
    • Type getRawType()：返回参数化类型中的原始类型，例如List 的原始类型为List
    • Type[] getActualTypeArguments()：获取参数化类型的类型变量或是实际类型列表，例如Map<Integer, String> 的实际泛型列表Integer和String 。需要注意的是，该列表的元素类型都是Type，也就是说可能存在多层嵌套的情况。
    • Type getOwnerType():返回是类型所属的类型，例如存在A类，其中定义了内部类lnnerA，则InnerA所属的类型为A，如果是顶层类型则返回null。这种关系比较常见的示例是Map<K,V>接口与Map.Entry<K,V>接口，Map<K,V>接口是Map.Entry<K,V>接口的所有者。
  • TypeVariable:表示的是类型变量，它用来反映在JVM编译该泛型前的信息。例如List中的T就是类型变量，它在编译时需被转换为一个具体的类型后才能正常使用。接口方法：
    • Type[] getBounds():获取类型变量的上边界，如果未明确声明上边界则默认为Object。例如class Test 中K的上界就是Person
    • D getGenericDeclaration():获取声明该类型变量的原始类型，例如class Test<K extends Person＞中的原始类型是Test 。
    • String getName():获取在源码中定义时的名字，上例中为K
  • GenericArrayType:表示的是数组类型且组成元素是ParameterizedType或TypeVariable,例如List[]或T[]。该接口只有Type getGenericComponentType()一个方法，它返回数组的组成元素。
  • WildcardType:表示的是通配符泛型，例如？ extends Number 和？ super Integer。WildcardType接口有两个方法:
    • Type[] getUpperBounds:返回泛型变量的上界
    • Type[] getLowerBounds:返回泛型变量的下界

设计模式

发表于 2018-09-15 | 分类于 设计模式

什么是设计模式

我们在设计开发过程中，经常能听到身边的人提到设计模式，那么设计模式到底是什么呢？这里，我不去找比较官方的解释了，只说说自己的理解。
首先，设计模式是前辈经验的沉淀，是对常见问题模型所提出的一些比较好的、通用的解决办法（前人帮我们尝试了很多不同的路，进而总结出来的比较好的办法）。
其次，设计模式不是万能的，不是所有的地方都适合套用，设计模式是一把双刃剑，在适合的场景下使用，不要强行套用。
第三，掌握了设计模式，能让你看懂很多之前一直看不懂的代码，这里也可以把设计模式理解成一种通用的语言，人家通过这种语言说的话（指的是应用了某种设计模式写出来的代码），你只有掌握了这种语言才能听懂。
以下是引用的一段话：

  1. 设计模式（Design pattern）代表了最佳的实践，通常被有经验的面向对象的软件开发人员所采用。
  2. 设计模式是软件开发人员在软件开发过程中面临的一般问题的解决方案。这些解决方案是众多软件开发人员经过相当长的一段时间的试验和错误总结出来的。
  3. 设计模式是一套被反复使用、多数人知晓的、经过分类编目的、代码设计经验的总结。使用设计模式是为了可重用代码、让代码更容易被他人理解、保证代码可靠性。
  4. 设计模式不是一种方法和技术，而是一种思想
  5. 设计模式和具体的语言无关，学习设计模式就是要建立面向对象的思想，尽可能的面向接口编程，低耦合，高内聚，使设计的程序可复用
  6. 学习设计模式能够促进对面向对象思想的理解，反之亦然。它们相辅相成

设计原则

在实际的开发中，我们要想更深入的了解面向对象思想，就必须熟悉前人总结过的面向对象思想的设计原则，下面看看面向对象设计的六大原则

单一职责原则（Single Responsiblity Principle）

单一职责，顾名思义，也就是职责要单一，不要把所有的事情都扔给一个对象处理。比如有一家餐馆，服务员负责点菜，传菜员负责上菜，厨师负责炒菜，收银员负责收钱，各司其职餐馆才能正常运营，如果一个人包揽所有的活，那么顾客的体验一定是极差的，餐馆开不了几天就会倒闭。
其实单一职责原则就是开发人员经常说的”高内聚，低耦合”，也就是说，每个类应该只有一个职责，对外只能提供一种功能，而引起类变化的原因应该只有一个。
在设计模式中，所有的设计模式都遵循这一原则。

开闭原则（Open Closed Principle）

核心思想是：对扩展开放，对修改关闭。
也就是说，对已经使用的类的改动是通过增加代码进行的，而不是修改现有代码。
通常是通过继承的方式来满足开闭原则，只需要对原有的类进行扩展，而不是修改原有类的代码。

里式替换原则（Liskov Substitution Principle）

核心思想：在任何父类出现的地方都可以用它的子类来替代。
也就是说，同一个继承体系中的对象应该有共同的行为特征。

依赖倒转原则（Dependency Inversion Principle）

核心思想：要依赖于抽象和接口，不要依赖于具体实现。
其实就是说：在应用程序中，所有的类如果使用或依赖于其他的类，则应该依赖这些其他类的抽象类或者接口，而不是直接依赖这些其他类的具体类。
为了实现这一原则，就要求我们在编程的时候针对抽象类或者接口编程，而不是针对具体实现编程。

接口分离原则（Interface Segregation Principle）

核心思想：不应该强迫程序依赖它们不需要使用的方法。
其实就是说：一个接口不需要提供太多的行为，一个接口应该只提供一种对外的功能，不应该把所有的操作都封装到一个接口中。
是不是也有点单一职责的味道在里面。

迪米特原则（最少认知原则，Principle of Least Knowledge）

核心思想：一个对象应当对其他对象尽可能少的了解
其实就是说：降低各个对象之间的耦合，提高系统的可维护性。在模块之间应该只通过接口编程，而不理会模块的内部工作原理，它可以使各个模块耦合度降到最低，促进软件的复用

设计模式分类

总体来说，设计模式分为三类23种：

  1. 创建型（5种）：工厂模式、抽象工厂模式、单例模式、原型模式、构建者模式
  2. 结构型（7种）：适配器模式、装饰模式、代理模式、外观模式、桥接模式、组合模式、享元模式
  3. 行为型（11种）：模板方法模式、策略模式、观察者模式、中介者模式、状态模式、责任链模式、命令模式、迭代器模式、访问者模式、解释器模式、备忘录模式

以上是设计模式的概念性知识，后面将逐个解释

什么是JNDI

Mybatis-DataSource

打开源码中datasource包，我们可以看到下面分为三个子包，分别是jndi、pooled、unpooled。

使用过mybatis的人都知道，我们在配置xml文件时，已经见过他们了，此处dataSource可选的三种内建数据源类型就是他们（也就是 type=”[UNPOOLED|POOLED|JNDI]”），是不是好巧

下面是三种内建数据源的使用场景，点击查看详细介绍：
UNPOOLED 这个数据源的实现只是每次被请求时打开和关闭连接。虽然有点慢，但对于在数据库连接可用性方面没有太高要求的简单应用程序来说，是一个很好的选择。 不同的数据库在性能方面的表现也是不一样的，对于某些数据库来说，使用连接池并不重要，这个配置就很适合这种情形
POOLED 这种数据源的实现利用“池”的概念将 JDBC 连接对象组织起来，避免了创建新的连接实例时所必需的初始化和认证时间。 这是一种使得并发 Web 应用快速响应请求的流行处理方式
JNDI 这个数据源的实现是为了能在如 EJB 或应用服务器这类容器中使用，容器可以集中或在外部配置数据源，然后放置一个 JNDI 上下文的引用

进入方法后，首先new了一个与当前集合容量相同的数组r；第二步，遍历数组的同时通过迭代器获取元素，并放置到数组r中，思考下，这里为什么要循环数组进行赋值而不是遍历集合呢？假设外层通过迭代器循环遍历集合，当我们在进行处理时，有另一个线程向集合中插入了一个新的元素，此时集合的size已经超过了r的容量，当处理新插入的元素时，就会抛出数组越界异常，因此直接遍历数组就可以避免这种情况。再看if (! it.hasNext()) // fewer elements than expected这个判断，相信经过上面的分析，你已经知道为什么了，对了，因为在循环的时候，另一个线程也有可能从集合中删除元素，如果迭代过程中发现没有下一个元素了，那就截取r数组前i个元素返回。return it.hasNext() ? finishToArray(r, it) : r;最后，判断集合是否还有剩余元素，如果没有直接返回r，如果有则进入finishToArray方法。下面来看看finishToArray方法

这里，首先取到数组的长度给i，循环体内又取数组长度给到cap，看看这个变量名在加上之前数组已经被填满了，二者一结合，就知道，嗯，应该是要扩容了。果然，当i和cap相等的时候，我们根据原数组长度计算出一个新的容量，新的容量为原来数组长度加上长度的一半再加1，为什么要加1呢，我是这么理解的，可能我们的容器中本来容纳的对象比较少，比如只有1个，此时进入这段代码进行扩容的时候，如果不加1，那1 + (1 >> 1)算出来还是1，再往下进行r[i++] = (T)it.next();操作的时候就会出错了。好了，我们继续往下看，我们发现多了个MAX_ARRAY_SIZE,这个东西的定义是这样的

粘这么多是希望多关注下上面的注释，看完你就明白了，因为部分虚拟机在数组的头部会预留一些信息，所以这个就是数组所能分配的最大容量了，如果试图分配更大的空间，可能会引发OutOfMemoryError。好了，回头再看，if判断是为了防止扩容后的容量超过MAX_ARRAY_SIZE了，如果真的超过了，就需要使用hugeCapacity方法重新计算扩容后的容量值了

进入hugeCapacity方法后（注意，传入的是原始容量+1），首先判断minCapacity是否小于0，旁边的overflow真实贴心，这说明minCapacity并不是太小，反而是太大，因为java中的int是有符号的，最高位是符号位，当一个数太大溢出了，表示的是符号位也就成了1，这个时候这个数就是负数了，此时，抛出OutOfMemoryError异常，如果minCapacity大于MAX_ARRAY_SIZE，那就把Integer.MAX_VALUE返回，否则返回MAX_ARRAY_SIZE。扩容完毕后，通过r[i++] = (T)it.next();赋值。继续循环，当容量满了之后，再次扩容。最终处理完成后,return (i == r.length) ? r : Arrays.copyOf(r, i);判断元素的实际个数和数组长度是否一致，一致则直接返回，不一致的话，通过Arrays.copyOf(r, i)返回刚好容纳所有元素的数组。
